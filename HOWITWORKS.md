# Принцип работы системы слежения RoboMaster S1

## Общая архитектура

Система состоит из четырёх логических слоёв:

1. **Слой связи с роботом** — SDK DJI RoboMaster, подключение по Wi-Fi точке доступа (режим AP)
2. **Слой компьютерного зрения** — детекция людей через YOLOv11 + трекинг через BoT-SORT
3. **Слой управления движением** — пересчёт ошибки позиции и расстояния в скорости колёс
4. **Слой интерфейса** — отображение состояния, лог, кнопки управления

---

## Подключение к роботу

При нажатии кнопки «Подключение» вызывается `_connect_to_robot()`:

- `ep_robot.initialize(conn_type="ap")` — подключение через Wi-Fi точку доступа робота
- Инициализируются подсистемы: `chassis`, `gimbal`, `sensor`
- Гимбал возвращается в центральное положение, затем наклоняется вниз на 15° (`moveto(pitch=15, yaw=0)`)
- Запускается подписка на TOF-дальномер: `sub_distance(freq=5, callback=_tof_callback)` — 5 Гц, данные в мм

## Компьютерное зрение: детекция и трекинг

### Модель детекции

Используется `yolo11n.pt` — нано-версия YOLOv11 из библиотеки Ultralytics.  
При запуске трекинга модель вызывается на каждом кадре:

```python
results = model.track(frame, persist=True, tracker="custom_track.yaml", classes=[0], conf=0.3)
```

- `classes=[0]` — только класс «человек»
- `conf=0.3` — порог уверенности детектора
- `persist=True` — трекер сохраняет состояние между кадрами

### Конфигурация трекера (`custom_track.yaml`)

Используется алгоритм **BoT-SORT** (`tracker_type: botsort`).

Ключевые параметры:
| Параметр | Значение | Смысл |
|---|---|---|
| `track_high_thresh` | 0.25 | Порог первичного сопоставления детекций с треками |
| `new_track_thresh` | 0.8 | Порог для создания нового трека (снижает ложные треки) |
| `track_buffer` | 300 | Кадров хранить «потерянный» трек (долго = устойчиво к окклюзии) |
| `match_thresh` | 0.6 | Порог сходства при сопоставлении (IoU + внешний вид) |
| `gmc_method` | sparseOptFlow | Компенсация глобального движения камеры через оптический поток |
| `with_reid` | False | ReID-модель отключена (экономия вычислений) |

### Поток обработки кадра (`_process_frame`)

1. Получить кадр с камеры: `ep_camera.read_cv2_image(strategy="newest")`
2. Запустить `model.track(...)` → получить боксы, ID треков, классы, уверенность
3. Для каждого человека с `conf > 0.3` — вычислить центр бокса `(center_x, center_y)`
4. Обновить словарь `self.tracks = {track_id: (cx, cy)}`
5. Нарисовать боксы (зелёный — не выбранная цель, красный — выбранная)

## Выбор цели

Цель выбирается кликом мыши по окну OpenCV (`_mouse_callback`).  
Находится ближайший трек в радиусе 100 пикселей от клика. Его ID записывается в `self.selected_person_id`.

---

## Управление движением

### Логика потери цели

- `last_seen_time[track_id]` — время последнего обнаружения трека
- Если выбранный трек отсутствует в `self.tracks` дольше `TRACK_TIMEOUT` (2 с) → цель считается потерянной
- При потере цели начинается режим **поиска**: вращение на месте со скоростью `SEARCH_ROTATION_SPEED` (20 ед/с)
- Если поиск длится дольше `SEARCH_TIMEOUT` (20 с) — цель сбрасывается, движение останавливается

### Расчёт скоростей колёс (`_calculate_movement_speeds`)

Два вида управляющего воздействия:

**1. Угловое (поворот)**  
Ошибка по горизонтали: `error_x = target_x - frame_width / 2`  
Линейно масштабируется в диапазон `[-MAX_ROTATION_SPEED, +MAX_ROTATION_SPEED]` (±80 ед/с).  
Если `|error_x| > frame_width / 4` → движение вперёд/назад блокируется (сначала довернуть).

**2. Линейное (вперёд/назад)**  
Зависит от расстояния по TOF-датчику:

| Расстояние | Поведение |
|---|---|
| < `MIN_DISTANCE_MM` (1500 мм) | Движение назад, скорость пропорциональна нарушению |
| `MIN_DISTANCE_MM` … `MAX_DISTANCE_MM` (1500–3000 мм) | Движение вперёд, скорость растёт пропорционально |
| > `MAX_DISTANCE_MM` (3000 мм) | Максимальная скорость вперёд |

**Сборка скоростей для омни-привода (4 колеса Mecanum)**:

```
w1 = -rotation + move   (переднее левое)
w2 =  rotation + move   (переднее правое)
w3 =  rotation + move   (заднее левое)
w4 = -rotation + move   (заднее правое)
```

### Ручное управление (клавиатура)

`pynput.keyboard.Listener` перехватывает нажатия глобально (вне фокуса окна):
- `W` / `S` — вперёд / назад (скорость 150 ед/с)
- `A` / `D` — стрейф влево / вправо (скорость 100 ед/с)
- Отпускание клавиши → остановка всех колёс

## Многопоточность

Основной поток — UI (Tkinter/pygame mainloop).  
При запуске трекинга создаётся `tracking_thread (daemon=True)`:
- читает кадры с камеры
- запускает детекцию/трекинг
- управляет колёсами
- отображает кадр через OpenCV

Остановка потока — через `threading.Event` (`stop_tracking_flag`): главный поток устанавливает флаг, поток трекинга проверяет его в цикле и завершается.

Состояние `self.tracks` обновляется только из потока трекинга. Чтение из главного потока (UI) не синхронизировано через мьютекс — это упрощение, допустимое для данного прототипа.

## Pygame-интерфейс (v0.2)

Окно 1280×720 разделено на две зоны:

- **Видеопанель (960×720, слева)** — кадры с камеры робота, преобразованные из BGR (OpenCV) в RGB (pygame) через `pygame.image.frombuffer`. Кадр масштабируется с сохранением пропорций и центрируется. Кликом по видеообласти выбирается цель.

- **Боковая панель (320×720, справа)** — индикаторы состояния, кнопки управления, лог событий.

Главный цикл работает в основном потоке (60 fps), трекинговый поток — daemon-поток (~30 fps). Кадр передаётся между потоками через `threading.Lock`.

## Запуск

```bash
source .venv/bin/activate
python main.py
```

Перед запуском робот должен быть включён, а компьютер подключён к его Wi-Fi точке доступа.
